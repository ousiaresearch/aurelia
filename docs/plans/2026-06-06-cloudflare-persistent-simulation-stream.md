# Cloudflare Persistent Simulation Stream Implementation Plan

> **For Hermes:** Use subagent-driven-development skill to implement this plan task-by-task.

**Goal:** Make Aurelia continuously observable and durable by streaming simulation deltas to a reliable Cloudflare-backed data plane, without forcing the Python simulation itself into Cloudflare Workers.

**Architecture:** Keep simulation compute outside Cloudflare for now, because Aurelia is Python + SQLite + long-running CPU work. Use Cloudflare as the ingestion, storage, replay, and public API layer: Workers receive signed event batches, Queues absorb bursts, D1 stores queryable summaries/events, R2 stores full DB/snapshot artifacts, Durable Objects provide single-writer live stream state and WebSocket/SSE fanout, and Pages hosts the dashboard.

**Tech Stack:** Python runner, Cloudflare Workers, Queues, D1, R2, Durable Objects, Pages, optional Hyperdrive/Postgres if D1 write/query shape becomes limiting.

---

## Decision Summary

### Do not run the current simulator directly inside Workers

Current Aurelia mechanics are not a good fit for Workers as the primary compute runtime:

- Source is Python, not JavaScript/TypeScript.
- The hot loop is SQLite + Python object mutation, not edge-friendly request handling.
- 200-year runs are minutes-to-hours, while Workers/Durable Object requests have CPU/request limits.
- Cloudflare Workers have no normal writable filesystem for live SQLite files.
- Long-lived, high-volume SQLite mutation should not be implemented as an always-open Worker loop.

Cloudflare should be the reliable data plane first, then optionally become compute later after a rewrite.

### Recommended first production shape

1. Python simulation runner remains on Colab/VM/Modal/Fly/Blackwell host.
2. Runner writes local world DBs as it does now.
3. Runner also emits compact JSON deltas every simulated year or every N monthly ticks.
4. Cloudflare Worker verifies HMAC and enqueues batches.
5. Queue consumer writes idempotently into D1.
6. Full `.db.zst` checkpoints and yearly snapshots go to R2.
7. Durable Object maintains current run head and fans out live updates.
8. Cloudflare Pages dashboard reads D1 + R2 manifest and subscribes to live stream.

This gives reliable persistence and live visibility without rewriting the simulator first.

## Cloudflare Product Mapping

### Cloudflare Pages

Use for:

- Aurelia live dashboard.
- Timeline browser.
- Per-world population/faction/discovery charts.
- Chronicle browser after Phase 2 generation.

Do not use for:

- Simulation compute.
- Runtime filesystem writes.
- Node-style `/api/*.js` serverless routes.

### Cloudflare Workers

Use for:

- `POST /ingest/run-event-batch` signed ingestion endpoint.
- `GET /api/runs/:run_id/status`.
- `GET /api/worlds/:world_id/timeline`.
- `GET /api/events?world=&year=&type=`.
- Authentication, rate limiting, validation.

Do not use for:

- Multi-hour Python simulation loop.
- Full DB mutation engine.

### Cloudflare Queues

Use for:

- Buffering event batches from the runner.
- Retrying transient D1/R2 failures.
- Decoupling runner speed from database write speed.

### Cloudflare D1

Use for queryable relational state:

- Run metadata.
- Yearly summaries.
- Event index.
- Current world state snapshots.
- Artifact manifests.

Avoid storing:

- Every NPC row on every tick.
- Full SQLite DB blobs.
- High-volume raw logs better suited for R2.

Cloudflare D1 paid limits observed from docs on 2026-06-06: 10GB maximum database size, 1TB storage/account, 50,000 databases/account, 1,000 queries per Worker invocation. This is enough for run summaries and indexed events, not for unbounded raw tick history forever.

### Cloudflare R2

Use for heavy artifacts:

- `world_dbs/{run_id}/{world}.db.zst`.
- `snapshots/{run_id}/snapshot_year_010.json`.
- `yearly_events/{run_id}.json`.
- Chronicles generated by Phase 2.

### Durable Objects

Use for single-writer live coordination:

- One Durable Object per active run.
- Keep current head: latest year, latest world summaries, connected dashboard clients.
- WebSocket/SSE fanout for live dashboard.
- Optional alarm-based heartbeat monitoring.

Do not use one Durable Object to hold all NPC state unless the sim is rewritten around DO storage and per-world single-writer semantics.

### Optional Postgres/Hyperdrive

Use instead of or alongside D1 if:

- You need complex analytical queries over millions of events.
- You want BI-style query tools.
- D1 write throughput or SQL features become limiting.

Cloudflare Hyperdrive can front Postgres while Workers remain the API layer.

## Data Model v1

### D1 tables

```sql
CREATE TABLE runs (
  run_id TEXT PRIMARY KEY,
  created_at INTEGER NOT NULL,
  source TEXT NOT NULL,
  mode TEXT NOT NULL,
  years INTEGER NOT NULL,
  worlds TEXT NOT NULL,
  npcs_per_world INTEGER NOT NULL,
  status TEXT NOT NULL,
  current_year INTEGER DEFAULT 0,
  metadata_json TEXT NOT NULL DEFAULT '{}'
);

CREATE TABLE world_years (
  run_id TEXT NOT NULL,
  world_id TEXT NOT NULL,
  year INTEGER NOT NULL,
  population INTEGER NOT NULL,
  births INTEGER NOT NULL DEFAULT 0,
  deaths INTEGER NOT NULL DEFAULT 0,
  factions INTEGER NOT NULL DEFAULT 0,
  discoveries INTEGER NOT NULL DEFAULT 0,
  great_persons INTEGER NOT NULL DEFAULT 0,
  migrations INTEGER NOT NULL DEFAULT 0,
  summary_json TEXT NOT NULL DEFAULT '{}',
  created_at INTEGER NOT NULL,
  PRIMARY KEY (run_id, world_id, year)
);

CREATE TABLE events (
  event_id TEXT PRIMARY KEY,
  run_id TEXT NOT NULL,
  world_id TEXT NOT NULL,
  year INTEGER NOT NULL,
  tick INTEGER,
  event_type TEXT NOT NULL,
  importance REAL,
  payload_json TEXT NOT NULL,
  created_at INTEGER NOT NULL
);

CREATE INDEX idx_events_run_world_year ON events(run_id, world_id, year);
CREATE INDEX idx_events_type ON events(event_type);

CREATE TABLE artifacts (
  artifact_id TEXT PRIMARY KEY,
  run_id TEXT NOT NULL,
  world_id TEXT,
  artifact_type TEXT NOT NULL,
  r2_key TEXT NOT NULL,
  size_bytes INTEGER,
  sha256 TEXT,
  created_at INTEGER NOT NULL
);
```

### Ingestion payload

```json
{
  "run_id": "aurelia-2026-06-06-fast-monthly-001",
  "sequence": 128,
  "mode": "fast-monthly",
  "worlds": {
    "solara": {
      "year": 42,
      "population": 24133,
      "births": 4810,
      "deaths": 4361,
      "factions": 3,
      "discoveries": 1,
      "great_persons": 0,
      "migrations": 77,
      "events": []
    }
  }
}
```

Idempotency key:

```text
run_id + sequence
```

Event key:

```text
sha256(run_id + world_id + year + event_type + canonical_payload)
```

## Implementation Tasks

### Task 1: Add runner-side Cloudflare emitter

**Objective:** Teach `colab/speed_run.py` to POST yearly summaries to a signed endpoint without blocking the simulation.

**Files:**

- Modify: `colab/speed_run.py`
- Create: `tests/test_cloudflare_emitter.py`

**Steps:**

1. Add CLI flags:
   - `--cf-ingest-url`
   - `--cf-ingest-secret-env`, default `AURELIA_CF_INGEST_SECRET`
   - `--cf-emit-every-years`, default `1`
   - `--cf-timeout`, default `10`
2. Add HMAC helper using `hmac`, `hashlib`, `json`.
3. Add non-fatal `emit_batch(payload)` with retry 3x and exponential backoff.
4. On failure, write payload to `output/pending_cloudflare_batches/*.json` for replay.
5. Test canonical JSON signature is stable.
6. Test failed POST writes pending batch.
7. Commit.

### Task 2: Create Cloudflare Worker ingestion service

**Objective:** Accept signed event batches, validate shape, and enqueue them.

**Files:**

- Create: `cloudflare/ingest-worker/package.json`
- Create: `cloudflare/ingest-worker/wrangler.toml`
- Create: `cloudflare/ingest-worker/src/index.ts`
- Create: `cloudflare/ingest-worker/test/index.test.ts`

**Steps:**

1. Define Worker route `POST /ingest/run-event-batch`.
2. Validate `X-Aurelia-Signature` HMAC.
3. Validate required fields: `run_id`, `sequence`, `worlds`.
4. Write raw payload to Queue.
5. Return `{ok:true, accepted:true}`.
6. Unit test invalid signature returns 401.
7. Unit test valid batch enqueues message.
8. Commit.

### Task 3: Add Queue consumer to write D1

**Objective:** Persist batches idempotently into D1.

**Files:**

- Modify: `cloudflare/ingest-worker/src/index.ts`
- Create: `cloudflare/ingest-worker/migrations/0001_init.sql`

**Steps:**

1. Create D1 schema above.
2. Add `processed_batches(run_id, sequence, processed_at)` table.
3. Queue consumer checks idempotency before processing.
4. Upsert `runs` and `world_years`.
5. Insert events with deterministic `event_id`.
6. On duplicate batch, ack without rewriting.
7. Commit.

### Task 4: Add R2 artifact upload path

**Objective:** Store full DB and snapshot artifacts in R2 while indexing metadata in D1.

**Files:**

- Modify: `colab/speed_run.py`
- Modify: `cloudflare/ingest-worker/src/index.ts`

**Steps:**

1. Add Worker route `POST /ingest/artifact-presign` or direct `PUT /artifact/:key` with HMAC.
2. Compress DBs with zstd/gzip in runner before upload.
3. Upload final DBs and `yearly_events.json` at run completion.
4. Worker writes `artifacts` rows with `sha256` and `size_bytes`.
5. Commit.

### Task 5: Add live dashboard API and stream

**Objective:** Expose current state and live updates.

**Files:**

- Create: `cloudflare/live-worker/src/index.ts`
- Create: `cloudflare/dashboard/` or integrate into existing Aurelia portal.

**Steps:**

1. Create Durable Object `RunLiveState` keyed by `run_id`.
2. On each processed batch, update DO with latest summaries.
3. Add `GET /api/runs/:run_id/status`.
4. Add `GET /api/runs/:run_id/events` backed by D1.
5. Add WebSocket/SSE endpoint for dashboard.
6. Build Pages dashboard with per-world cards.
7. Commit.

### Task 6: Add replay utility

**Objective:** Ensure no data loss if Cloudflare is temporarily unavailable.

**Files:**

- Create: `scripts/replay_cloudflare_batches.py`

**Steps:**

1. Read `pending_cloudflare_batches/*.json`.
2. Re-sign and POST each payload.
3. Move successful files to `replayed/`.
4. Leave failed files in place.
5. Commit.

## Acceptance Criteria

- A local/Colab run can continue even if Cloudflare is down.
- Every emitted yearly batch is either accepted by Cloudflare or saved locally for replay.
- D1 can answer: latest population per world, events by year/type, run completion status.
- R2 contains full final DBs and `yearly_events.json` for archival/reanalysis.
- Dashboard can show live run progress without reading Google Drive.
- Duplicate POSTs do not duplicate rows.
- No credentials are written to `.env` files in the repo.

## Future Phase: Cloudflare-native simulation

Only attempt after the streaming data plane is stable.

Possible paths:

1. TypeScript rewrite of core monthly mechanics; one Durable Object per world.
2. WASM/Pyodide experiment for a small subset of mechanics.
3. Cloudflare Containers if a long-running Python container deployment is acceptable and available for the account.
4. External compute forever, Cloudflare only as data/API layer.

The recommended path remains external compute + Cloudflare durable stream until the model stabilizes.
